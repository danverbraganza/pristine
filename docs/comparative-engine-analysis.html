Comparative engine analysis

Configuration patterns from Unity, WebKit, Unreal, Godot, and VS Code — and which survive translation to Pristine.

Pristine harness engine · pre-blueprint research for the Configuration File roadmap item

Framing

Pristine is positioned as a harness engine: not an agent, not a UI, not a product, but the core that those layers are built on. The analogy is Unity to games or WebKit to browsers. That positioning sets the bar for the configuration story: deep, structured, evolvable over many years, and never the engine's master.

This document surveys how four mature engines (and one editor for contrast) handle configuration, then distills the patterns that translate to Pristine and the ones that do not.

Existing project material

The Configuration File phase is already anchored in repo docs. Three excerpts:

DESIGN.md (roadmap item 1): Replace the hardcoded HarnessBuilder setup with file-driven configuration. A configuration file will control how many Agents to run, their system prompts, which tools are available to each Agent, model assignments via ModelRole, and inter-agent routing rules. The builder remains available for programmatic use; the config file is sugar that produces the same builder calls.

DESIGN.md (philosophy): Configuration is the primary user interface. Rather than hard-coding agent topologies, Pristine will support declarative configuration files that describe which agents to run, how they are wired together, and what capabilities each has. The builder API remains the programmatic escape hatch for cases configuration cannot express.

ARCHITECTURE.md (Harness): Initially, the Harness will be constructed via a builder function pattern, but the eventual goal is going to be for the Harness to be customized via a configuration file, which will control how many agents to run, any custom prompts, what tools they have and how they can interact with each other.

The hardcoded surface being replaced lives in src/lib.rs::run_async (lines 63–113):

  • One Anthropic model, name from --model CLI flag, key from ANTHROPIC_API_KEY env.
  • One agent with hardcoded SYSTEM_PROMPT, single ModelRole::Default mapping.
  • Hardcoded tool set: AddTool, ExecBash, Edit, Read, Write, Insert — every agent gets every tool.
  • No routing; one agent; stdio JSON-RPC client.

HarnessBuilder (src/harness.rs:99) is the programmatic API the config layer must produce calls into: add_model(ModelId, Arc<dyn ARModel>), add_agent(PendingAgent { id, system_prompt, model_id }), add_tool(Arc<dyn Tool>), build().

Case studies

1. Unity — composition-first, sharded, ID-stable

  • ProjectSettings/ is not one file; it is roughly 30 YAML files, each scoped to one subsystem (Audio, Graphics, Input, Physics, Quality, Tags, Time, Editor…). Each carries a serialization version key.
  • ScriptableObjects = config-as-data: typed assets authored once and referenced by components. The Pristine analog would be tool configurations, system prompts, ModelRole bindings — named, typed, reusable.
  • GUIDs over names. Every asset has a stable GUID; renames don't break references. The .meta sidecar file exists for this alone.
  • Composition by attachment. A GameObject is a list of Components, each with serialized fields. Topology is the config; there is no separate "wiring" file.
  • Per-target overlays. PlayerSettings has per-platform sections; QualitySettings has tiers. The same logical config evaluates differently per target.

2. WebKit — typed schema, codegen, embedder-owned

  • Settings.yaml is the source of truth: roughly 800 preferences, each declared with type, default, per-port override, and lifecycle tier. A build script generates C++ accessors, defaults table, and the IPC marshalling. Hand-written settings code does not exist.
  • Layered tiers: build-time FEATURE_DEFINES, runtime defaults, runtime overrides via WKPreferences, experimental flags via WebPreferencesExperimental.yaml. Each tier has different mutability and audit semantics.
  • Embedder owns the top level. WKWebViewConfiguration, WKProcessPool, WKPreferences are config objects the embedder constructs and hands to the engine. There is no file the engine reads on its own — config flows in through the API. The file/schema exists for the embedder's tooling, not the engine's runtime.
  • Per-context, not global. Each WebView has its own configuration; sharing happens by sharing config objects, not by global state.

3. Unreal — three-tier INI layering, partial hot reload

  • Base*.ini (engine defaults) → Default*.ini (project overrides, in-repo) → User*.ini (per-developer, gitignored). Three layers, deterministic precedence.
  • Per-platform overlays: Windows/WindowsEngine.ini overrides DefaultEngine.ini only on that target.
  • cvars are runtime experimental knobs, exposed via the console and persisted to user config if pinned. Distinct from settings; closer to feature flags.
  • Most keys are restart-required; a documented subset hot-reloads.

4. Godot / VS Code — the lightweight end

  • Godot: project.godot is a single INI-ish manifest; scenes (.tscn) carry their own data. Plugins register via plugin.cfg.
  • VS Code: layered JSON — default → user → workspace → folder. Schema-defined; extensions contribute settings via contributes.configuration in package.json.

What survives translation to Pristine

PatternSourceVerdictWhy
Config file produces builder calls, never owns the runtime DESIGN.md already + WebKit embedder model Yes — load-bearing. Keeps SDK-first posture intact. Config is one of several front-ends.
Schema-as-source-of-truth, codegen the rest WebKit Partially, deferred. Premature at v1; commit to a single pristine-config module now, add schemars-driven JSON Schema later.
Subsystem sharding (multiple files, top-level sections) Unity, Unreal Yes — as sections, not files, at v1. One pristine.toml with [harness], [[models]], [[agents]], [[tools]], [[routes]]. File sharding via include = […] can come later without breaking schema.
Three-tier layering: shipped defaults → project → user Unreal, VS Code Yes. Solves secrets (user tier is gitignored), solves "what does pristine do with zero config" (shipped defaults supply a single-agent harness), keeps the builder-only path alive (user-supplied builder == top tier).
Stable IDs, not names, for references Unity GUIDs Adopt the spirit, not the cost. ModelId and AgentId already exist. Config references must use these. No .meta-file overhead — the name in TOML is the id (models.anthropic-default keys the table). Renames break references loudly during validation.
Per-target / per-environment overlays Unity, Unreal Reserve, don't build. A [profile.dev] / [profile.prod] namespace can be left out at v1 but the schema shouldn't preclude it.
Env-var indirection for secrets (${ANTHROPIC_API_KEY}) Universal Yes. The repo already pulls the key from env; the config layer must preserve that without inviting plaintext keys into files.
Typed validation phases (parse → resolve → semantic-validate → build) WebKit codegen, every game-engine loader Yes. Each phase has its own typed errors with span info. Bad configs must fail clearly and early.
Unknown-keys = warn, not error Unreal, Unity Yes. Long-lived engines outlive their schemas; forward-compat by ignoring unknown sections with a warning.
Forward-compat slots for roadmap items (discipline) Yes. Reserve [[skills]] and [[routes]] keys in the v1 schema even though feature work lands later — squatting on the namespace now avoids breaking changes later.
Hot reload WebKit, Unity Defer. Restart-only at v1. But Harness owns the resolved config — no global static — so hot-reload remains a future option.

The one shape-defining call

The deepest pattern across all four engines, and the one Pristine should commit to first:

The engine never reads config files. The engine consumes typed config objects through its construction API. Files exist for the embedder's tooling layer, not the engine's runtime.

Concretely for Pristine: Harness and HarnessBuilder know nothing about TOML, file paths, env vars, or layering. There is a separate pristine-config (module or crate) whose job is file → builder calls, and a separate concern again for "the pristine run binary's policy for finding and layering files." This keeps the engine-as-library-engine posture and matches WebKit's split exactly.

Decisions (round 1)

Captured from a vision-holder pass over the survives-translation table. Each item closes (or sharpens) one or more of the open questions below.

Schema source of truth: Rust structs, not codegen

The WebKit YAML-plus-codegen pattern is rejected as not Rust-native. pristine-config will define pub struct Config { … } with #[derive(Deserialize)]; TOML is one serde front-end among potentially several (JSON Schema export later via a schemars derive remains an option, still no build-step codegen).

Crate boundary: separate library

pristine-config is a separate library invoked from the binary's main. Harness and HarnessBuilder know nothing about TOML, file paths, env vars, or templating. File-to-builder translation is the library's sole responsibility.

Subsystem sharding by sections

One file, top-level sections. File-level sharding (multiple files with include = […]) is deferred — not needed at v1.

Layering / discovery: two-mode, no merging

  • No CLI flag → read shipped default.toml (a single-agent vanilla coding agent).
  • 1p -c <file> / --config <file> → that file is used entirely. No fallback to default.toml. No separate user-tier overlay.

The three-tier Unreal/VS Code layering pattern is rejected. There is no merging behavior to document and no precedence rules to debug.

Unknown keys: error, do not limp

#[serde(deny_unknown_fields)] on every config struct. A typo in a key name or an obsolete field surfaces as a load-time error with the key name and line. Failing loudly beats silently running a different topology than the one the operator authored.

Secrets: {{ENV_VAR}} templating, resolved at config-load time

The TOML carries placeholders like api_key = "{{ANTHROPIC_API_KEY}}". pristine-config substitutes from the process environment as it loads. Missing env vars are a hard error.

Engineering note (no decision required yet): the substitution should walk the parsed toml::Value tree rather than mutate the raw text, to avoid TOML-injection from env-var contents containing quotes or backslashes. The user-visible behavior is text replacement either way.

--model CLI flag goes away

Model name lives in default.toml (or the supplied config). One source of truth; no flag-vs-file precedence to reason about.

No reserved forward-looking slots

Skills, routes, profiles, hot-reload — none get placeholder sections in the v1 schema. They are added when their feature work lands. The smallness of the team makes coordinated schema extensions cheap, and avoiding speculative slots avoids the cost of designing the wrong shape twice.

Hot reload: deferred

Harness owns the resolved config (no global static), so adding hot-reload later remains an option, but it is not built in v1.

Stable IDs vs names: TBD

No decision. Names are the identifiers for now; the question is left open pending evidence (rename-pain, cross-file references, etc.).

Open questions remaining for the blueprint phase

  • File format. TOML is now the working assumption (default.toml shipped, rust-bucket.toml precedent, Rust-ecosystem default). Flagging for the blueprint to confirm or contradict.
  • Tool wiring shape. Enumerate per-agent tool lists vs. global registry + per-agent allowlist. Affects both the schema and the HarnessBuilder ↔ Agent wiring path.
  • ModelRole keying. Free-form strings vs. enum-with-extension-point. Today only Default exists; future Summarizer/Critic etc. arrive without warning.
  • Stable IDs vs names (carryover from round 1).
  • Templating implementation. Raw-text substitution before TOML parse vs. toml::Value-tree walk after parse. Same user-visible behavior; different failure surface.
  • Per-agent tool configuration data. If a tool ever takes constructor parameters (allowlists, audit hooks, etc.), the config schema needs a shape for that. Today's built-ins all take a bare new(); this is preview-only until a tool needs it.
